## 🚨 最重要ルール（常に遵守）

### 1. shadcn/ui 使用必須

**すべての UI コンポーネントは shadcn/ui を使用すること**

- 新しい UI コンポーネントが必要な場合、まず shadcn/ui レジストリを確認
- 利用可能なコンポーネントがあれば、必ず shadcn/ui を使用
- 生の HTML 要素（`<button>`, `<input>` など）の直接使用は禁止

```bash
# コンポーネント追加コマンド
pnpx shadcn@latest add <component>
```

❌ **禁止**: `<button className="...">クリック</button>`  
✅ **正しい**: `<Button variant="default">クリック</Button>`

### 2. TypeScript 厳格モード

- `any` の使用は原則禁止
- 不明な値は `unknown` を使用
- 外部入力は Zod でバリデーション

### 3. ビルドテスト必須

**コード実装後は必ず以下を実行:**

```bash
pnpm build
```

- エラーが発生した場合は即座に修正
- ビルドが成功するまで作業を完了としない
- 最大 3 回まで自動修正を試行

### 4. 実装範囲の制限

- 1 回の実装は必ず小さく分割
- 1 つの機能、1 つのコンポーネント、1 つの関数のみを実装
- コードの各行が何をしているかを明確に説明できるレベルで実装

### 5. ブランチ管理

- 開発作業開始前に現在のブランチを確認
- main ブランチにいる場合は作業用ブランチを作成して移動
- ブランチ名: `feat/add-exercise-modal`, `fix/history-calendar-bug` など

## 🔧 コーディング原則

### 設計原則

- **関心の分離**: 影響範囲を限定する
- **高凝集**: 内部変更を容易にする
- **疎結合**: 依存関係を減らす
- **シンプルを最優先**: オーバーエンジニアリング禁止
- **最小限の設計・実装**: 常に必要最小限で実装する

### useEffect 使用ポリシー

useEffect は外部世界との同期のためだけに使用すること。

**許可される用途**: API 呼び出し、WebSocket 接続、ブラウザ API、外部ストアの購読、タイマー

**禁止されるアンチパターン**:

- props や派生値をローカル state にコピー
- フラグの変更に応じてロジックを実行
- イベントハンドラの代わりに effect 内でユーザーアクションを処理
- 空の依存配列で一度だけ初期化（代わりに useMemo を使用）

## 📝 コーディングスタイル

- **関心の分離**: 再利用可能なコンポーネント設計
- **App Router 規約**: `page.tsx`, `layout.tsx`, `loading.tsx`, `error.tsx` などに従う
- **TypeScript**: 厳格な型定義、`any` の使用禁止
- **Tailwind**: ユーティリティファースト、インラインスタイルは避ける
- **モバイルファースト**: スマホアプリ風の UI 設計

### 読みやすいコードの原則

- **明示的な命名**: 変数名・関数名は処理内容が一目で分かるように命名
- **コメントの充実**: 複雑な処理には必ずコメントを記述
- **1 つの責任**: 1 つの関数・コンポーネントは 1 つの責任のみを持つ
- **適切な分割**: 長い関数（20 行以上）は分割を検討

### Supabase MCP の使用

データベース情報を確認する際は、Supabase MCP サーバーを使用してください。

詳細は `.cursor\rules\supabase.mdc` を参照してください。

## 💾 Git コミット規約

### 基本フォーマット

```
<Prefix>: <サマリ（命令形/簡潔に）>

- 変更内容1（箇条書き）
- 変更内容2（箇条書き）

Refs: #<Issue番号>（任意）
```

### Prefix一覧

- `feat`: 新機能の追加
- `fix`: バグ修正
- `refactor`: リファクタリング
- `perf`: パフォーマンス改善
- `test`: テスト追加/修正
- `docs`: ドキュメント更新
- `build`: ビルド/依存関係の変更
- `chore`: 雑務（ツール設定等）

### コミット粒度

- **原則**: 1 コミット = 1 意図
- **変更ファイルが複数ある場合**: 原則として「ファイル単位」で個別コミット
- テスト/ビルドが通る状態を保ち、不要変更は含めない

詳細は `.cursor\rules\commit-message-format.mdc` を参照してください。
