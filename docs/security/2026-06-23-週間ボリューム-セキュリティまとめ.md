# 週間ボリュームサマリーのセキュリティまとめ

## 1. 概要

週間ボリュームサマリー機能では、`src/lib/actions/weekly-summary.ts` の `getWeeklySummary(userId: string)` が週間トレーニング量を DB から集計します。自動セキュリティレビューで、このサーバーアクションに IDOR（Broken Access Control）の可能性が HIGH として指摘されました。

## 2. どんなセキュリティ課題か

IDOR は、ユーザーが本来アクセスできない他人のデータを、ID を差し替えるだけで取得できてしまう問題です。

今回の `getWeeklySummary(userId)` は `"use server"` のサーバーアクションです。サーバーアクションは通常の画面操作だけでなく、クライアントから直接 POST 到達し得るエントリポイントです。そのため、引数の `userId` をそのまま信頼して DB を読むと、悪意あるクライアントが他人の `userId` を渡した場合に、その人の週間サマリーを取得できる可能性があります。

## 3. 起こりうる被害

- 他人の週間トレーニング量を見られる可能性があります。
- 具体的には、総ボリューム、セット数、ジム回数などです。
- ただし、露出する情報は週間トレーニング量であり、PII や認証情報ではありません。
- そのため、自動レビューの HIGH 判定はやや過大で、実害は中程度以下と評価しています。

## 4. どう対応したか

今回の指摘は、`getWeeklySummary` だけに新しく発生した欠陥というより、既存サーバーアクション群に共通する設計パターンです。

既存の `getTotalWorkoutDays`、`getExercises`、`getBig3TargetValues`、`getUserMaxWeights` など、約 14 個のサーバーアクションも `userId` を引数で受け取り、それを使って DB にアクセスしています。新しい `getWeeklySummary` は、この既存慣習に合わせて実装されています。

正規の呼び出し元である `src/app/(app)/page.tsx` では、`userId` はサーバーセッション `getAuthSession()` から取得されているため安全です。

自動レビューと Codex のセカンドオピニオンでは、どちらも「今回の PR でこの 1 アクションだけ直す」のではなく、「既存パターンを維持し、全サーバーアクションを対象にした横断ハードニングを別課題化する」判断を推奨しました。

そのため、今回は意図的に `getWeeklySummary` を据え置き、恒久対策は `docs/security/server-action-authz-followup.md` に別課題として記録しました。

## 5. 対応した結果どうなったか

- 週間ボリュームサマリー機能は、既存の設計パターンと整合した状態でリリース可能です。
- IDOR の可能性は未修正ですが、リスクは把握済みで文書化されています。
- 露出データの機微度は低く、実害は中程度以下と評価しています。
- 恒久対策の方針は具体化済みで、全サーバーアクションを対象にした別課題として扱います。

## 6. 恒久対策

別課題では、サーバーアクション単位で以下の方針に移行します。

- 各アクション内で `getAuthSession()` を呼び、セッションから `userId` を導出する。
- 未認証の場合は処理を拒否する。
- 互換性のために `userId` 引数を残す場合は、`userId === session.user.id` を検証する。
- 共有ロジックは `server-only` な DAL に分離し、サーバー側からのみ呼び出す。
- 読み取り、書き込み、ゲスト利用（`null` userId 許容）が混在するため、一括置換ではなくアクション単位で移行する。

想定する基本形は以下です。

```ts
const session = await getAuthSession();
const userId = session?.user?.id;
if (!userId) throw new Error("UNAUTHENTICATED");
```
